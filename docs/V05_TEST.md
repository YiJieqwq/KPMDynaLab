# v0.5 process-session test

## New behavior

- CLI forks the target in a blocked state.
- CLI registers `TARGET <pid>` with KPM before releasing the child.
- KPM seals the selected profile before target `exec`.
- `wake_up_new_task`, `do_execveat_common`, and `do_exit` generate FORK/EXEC/EXIT events.
- Tagged descendants inherit the analysis session ID.
- Block events carry session ID and parent PID when generated by a tagged subject.
- Each run receives `/data/local/tmp/dynalab-run-<pid>` through `DYNALAB_WORKDIR`.
- After the target tree becomes inactive, CLI asks whether to remove that work directory.
- If descendants remain active, cleanup is skipped and the session stays SEALED.

## Install

Use matching v0.5 artifacts; the binary event ABI changed from v0.4.

```sh
kpatch kpm unload KPMDynaLab
kpatch kpm load /data/local/tmp/KPMDynaLab-0.5.0-test.kpm
chmod 755 /data/local/tmp/dynalab
chmod 755 /data/local/tmp/device_smoke_sfx.sh
```

Reconfigure the login verifier through trusted Manager CTL0.

## Test

```text
DynaLab> profile auto
OK PROFILE
DynaLab> clear
OK CLEAR
DynaLab> run /data/local/tmp/device_smoke_sfx.sh
OK TARGET
OK SEALED
```

The sample performs:

```text
SFX launcher
  -> extracts .stage1.sh
  -> exec stage1
       -> creates .stage2.sh
       -> exec stage2
            -> creates payload.note and loop backing image
            -> dd writes loop block device
```

Expected event outline:

```text
EXEC  target-pid ... /data/local/tmp/device_smoke_sfx.sh
EXEC  same-pid   ... /system/bin/sh or .stage1.sh
EXEC  same-pid   ... .stage2.sh
FORK  dd/sha256/losetup children
BLOCK_WRITE ... session=<nonzero> SIMULATE
EXIT  tagged processes
```

Shell interpreters may cause several EXEC/FORK events, and short-lived utility processes can appear between the stages.

After completion:

```text
Cleanup session workdir ...? [Y/n]
```

Press Enter or `y` to remove extracted artifacts. Enter `n` to retain them for inspection.

## Limitations

- Subject storage and event storage are fixed-size, lockless prototype arrays.
- Target/descendant tracking is PID-based and not yet hardened against PID reuse.
- Cleanup only removes the dedicated `DYNALAB_WORKDIR`; it never guesses and deletes arbitrary files dropped elsewhere.
- A background descendant keeps the session SEALED and suppresses automatic cleanup. Use `active` and inspect events before deciding how to terminate it.
- Published builds still expose procfs and dmesg diagnostics.
